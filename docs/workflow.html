1Create a Project

A project is your workspace. Give it a name and — most importantly — a description. The description is sent to the AI on every generation, so the model stays grounded in your domain rules, compliance constraints, and product context.

💡 Example description
"HIPAA-regulated patient portal — every action must log to an audit trail. User roles: Patient, Provider, Admin."

You can also generate the description automatically by uploading a PRD or product-overview document.

2Add a Feature

Features group test cases inside a project. Add one per UI flow you want to test:

  • Login
  • Checkout
  • User profile
  • Search & filter

Features are lightweight — name only. The interesting metadata lives on the test cases and the generation history.

3Provide Input

Pick one or more input sources. Multi-input is the norm — Jira ticket plus a Figma frame plus a screenshot is a common pattern.

How the AI weights each source:

SourceBest forHow the AI uses it
📋 Jira Acceptance criteria, PRDs in tickets Trusts the issue is fully scoped — tests every AC, no scope drift
🎨 Figma UI specs with prototype interactions Uses element names verbatim (devs build close to design = real selectors)
📷 Screenshot The actual built app Treats every visible label as a real Playwright target
✏️ Text Quick PRDs, inline requirements Tests exactly what's described, nothing more
ℹ️ Multi-source rule of thumb: Jira / text describe what the feature must do. Figma / screenshot describe how it looks. Combine them for the best coverage.

The "critical-path first" ordering

The AI is told to think like a senior QA lead: "What is the ONE test that, if it failed, would mean the feature is broken end-to-end?" That test becomes index 0 of the array. The rest follow in descending criticality:

1
Critical happy path
valid creds → Sign in → verify logged in
2
Critical negative paths
wrong password → error shown
3
Secondary user journeys
forgot password, remember me, sign-up nav
4
UI verification
labels, placeholders, button states, colors
5
Edge cases
max length, special chars, rapid clicks
🎚 Type filter? If you filter to a single type (e.g. smoke only), the AI still orders within that filter by criticality. The first smoke test for a login feature is the visibility of the Sign in button — not the brand logo.

📋 Test Cases tab

Flat list. Each row has title, steps, expected result, priority, type. Edit inline, delete, multi-select for bulk operations.

🕒 Generations tab

Read-only audit trail. Each card = one generate or iterate run. Shows:

  • When — timestamp
  • Trigger — generate or iterate
  • Inputs — Jira URL / Figma URL / screenshot thumbnails / text snippets you submitted
  • Outputs — the test cases this run produced (click a title to jump to it)

When you delete all test cases from a generation, its card disappears automatically (soft hide — DB row remains).

♻️ Iterate

Click Iterate on a feature → describe what to add ("add more edge cases for password rules") → the AI generates non-duplicate tests scoped to your instruction. Same priority-first ordering applies within your filter.

▶️ Run with Playwright (optional)

For any test case, hit Run to send it to the Playwright runner. The app converts your manual steps into Python code (page.get_by_role(...), etc.) and executes it against a base URL. Last-run status (✅ passed / ❌ failed / ⚠️ error) is shown on the test row.

  • Frontend is a single HTML page + single app.js. No build step, no framework.
  • Backend is FastAPI with async SQLite. Generation runs on the request thread; LLM calls use asyncio.to_thread where the provider SDK is synchronous (Gemini).
  • LLM provider is chosen per-call from the UI or settings. API keys are stored only in local SQLite — never read from .env.
